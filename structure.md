## Глава 3: Зачем это нужно?

- Единый проект со сквозным поиском вместо разрозненных источников
- Markdown: прост в освоении, низкий порог входа, широко поддерживается современными инструментами
- OpenAPI — машиночитаемый стандарт описания API. Вокруг него — целая экосистема готовых инструментов (tools.openapis.org): визуализация, тестирование, генерация кода — всё «из коробки»
- Интерактивная документация (Redoc, Swagger UI) — живой интерфейс с поиском, фильтрацией, примерами запросов
- Автоматическая проверка: инструменты сверяют реальное поведение сервера с документацией — ошибки ловятся до того, как их найдёт пользователь
- Генерация клиентского кода: по описанию API автоматически создаются готовые функции для фронтенда — разработчику не нужно писать их вручную
- Единый формат, понятный всем — от фронтенда до QA

## Глава 6: Чем это обеспечено

- Утилита open_api_generator.py — 2 394 строки Python-кода
- Генерирует OpenAPI-спецификацию из существующих исходников проекта, не требуя изменений от разработчиков
- Парсит 5 типов источников: RST-документацию, описания функций в коде, файлы маршрутов, примеры HTTP-запросов, конфигурацию проекта
- На выходе — готовый файл для Redoc / Swagger UI

## Глава 7: Реализации парсера - вводные

- Огромный объём документации, копившейся годами: два проекта, 272 эндпоинта API (ostecPlatform — 147, smartLine — 125)
- Полное отсутствие единого стиля оформления — за годы разные разработчики описывали API по-разному:
  - где-то параметры — маркированный список (smtAlerts.py, slProducts.py)
  - где-то — таблицы с колонками «Атрибут / Описание / Обязательный» (model.rst, productModelPrinter.rst)
  - где-то — просто пример JSON без пояснений (smtObjects.py)
  - разные форматы описания параметров: структурированный Sphinx-формат vs свободный текст
  - даже стиль кавычек в коде отличается от файла к файлу
- Существенные структурные различия между проектами ostecPlatform и smartLine:
  - конфигурация в разных папках: config/ vs conf/
  - в одном проекте один файл конфигурации, в другом — 13 отдельных файлов для каждого сервера
  - формат описания маршрутов различается: простая строка vs массив с указанием HTTP-методов
  - в одном проекте таблицы для описания моделей данных, в другом их нет вообще
- Утилита должна распознавать и корректно обрабатывать все эти варианты

## Глава 8: А если попробовать без ИИ?

- Нет готовых решений, адаптированных под наши вводные — утилиту придётся писать с нуля
- Большой объём кропотливой работы с низкоуровневым кодом разбора текста
- Необходимость учёта и обработки множества нестандартных случаев
- Результат тяжело поддерживать: даже с детальной документацией любые изменения в исходниках могут сломать парсер

## Глава 10: R&D / НИОКР

- «СТЗ» — в части алгоритмов справедливо то же, что и для парсера:
  - нет готовых рецептов под наши вводные
  - большой объём кропотливой работы
  - множество нестандартных случаев
  - часть потенциальных проблем возможно ещё не осознана
- ИИ не гарантирует 100% успех в уникальных алгоритмах, но помогает:
  - быстро прототипировать и проверять гипотезы — вместо дней на один прототип можно попробовать несколько подходов за часы
  - находить ошибки в логике на ранних этапах: агент анализирует код и указывает на граничные случаи, которые разработчик может пропустить
  - изучать предметную область: ИИ обобщает научные статьи, документацию и примеры, ускоряя погружение в тему
  - генерировать тесты и отладочные сценарии, чтобы быстрее убедиться в корректности решения

## Глава 11: Стандартные проекты

## Глава 12: Безопасность

- Код не используется для обучения моделей на платных планах (Pro, Max, Team, Enterprise) — если не дать на это явное согласие в настройках
- Для Team и Enterprise планов данные защищены коммерческими условиями — Anthropic не использует их для обучения ни при каких обстоятельствах
- Хранение переписки: 30 дней, если обучение на данных отключено; до 5 лет, если пользователь сам разрешил использовать данные для обучения
- Удалённый диалог не используется для будущего обучения
- Настройки приватности: claude.ai/settings/data-privacy-controls

## Глава 14: Дополнение — RAG

- RAG (Retrieval-Augmented Generation) — единое окно доступа ко всем знаниям компании через ИИ
- Все данные хранятся во внутреннем контуре, ничего не отправляется третьим лицам
- Доступ разграничивается — сотрудник видит только те документы, к которым у него есть права
- Роль ИИ:
  - синтез — собирает ответ из нескольких документов в один связный текст. Ответ соответствует намерениям пользователя, а не просто набору ключевых слов из запроса
  - диалог — можно уточнять, переспрашивать, углублять вопрос
- Снижение bus factor / «фактора кирпича»: количество экспертов — носителей незафиксированных знаний — потеря которых ведёт к невозможности продолжения работы. RAG делает знания из документов доступными всем, а не только тем, кто «знает, где искать» или «знает, кого спросить»
